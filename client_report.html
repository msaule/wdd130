Final Project

Client, lead developer, and junior developer report.

This report reflects on the different roles I had during the final team project. I worked as a client for my own analytics portfolio site, lead developer for Mason Anderson's site, and junior developer on two other class sites.

Client Report

For my personal site plan, I chose content and images based on the analytics portfolio idea I had already presented in my elevator pitch. I wanted the site to show who I am, what I can build, and how my projects connect to business intelligence, statistics, data science, and machine learning. I picked content that would make sense for recruiters, internship managers, professors, or anyone else trying to quickly understand my work.

I tried to make the site plan practical for the developers by including page purposes, written content, image ideas, assets, colors, fonts, and wireframes. After receiving feedback that the homepage should not be overloaded with technical details, I planned it around a simple introduction, a few strong projects, and clear links to deeper information.

I communicated with my lead developer, Eason Platt, through class/team messages. I sent over the site plan and materials, then checked in during the project to ask if the developers needed anything else from me. They said they did not need anything else at that point, so I waited for updates. The main communication issue was that the published site existed, but it had not been updated with my actual content yet when I checked it.

My developer team had the correct materials from me, but the site did not fully meet my expectations by the time I reviewed it because my content had not been added yet. I understand that team projects can be hard to coordinate, but as the client I would have liked to see earlier progress and a clearer timeline for when the homepage and child pages would be updated.

Next time I hire a team for a project, I would set more specific check-in dates and ask for small progress updates earlier instead of waiting until the end. I would also make a short priority list so the developers know which sections matter most if they run short on time.

Lead Developer Report

As lead developer, I worked on Mason Anderson's Local with Lance site. My main responsibility was to build and style the homepage, create the GitHub repository, publish the site with GitHub Pages, and coordinate with the junior developers who were assigned the Housing and Social Media pages.

I managed the team by giving the junior developers their assigned pages and reminding them to work from the shared structure and stylesheet. One important lesson was how easy it is for a teammate to accidentally push older code over newer work. After that happened once, I made sure to tell the team to pull the latest version before editing and to ask before pushing if something looked wrong.

I communicated with the client by working from the site plan and making choices that matched the requested style, colors, fonts, logo, and page layout. I also improved the site images because the earlier images were too low quality and made the page feel unfinished. The client wanted better visuals, so I added stronger local images and used them across the homepage and child pages.

Publishing the site with GitHub Pages was a useful part of the project. I set up the repository, pushed the homepage and shared styles, checked the published URL, and used the live site to catch issues that were not as obvious locally. Managing the site with GitHub was sometimes stressful because of overlapping teammate changes, but it also made the project feel closer to real development work where everyone has to coordinate before pushing.

Junior Developer Report

As a junior developer, I worked on two team sites. For Project 5, I worked under Breanna Caballero Salazar on Sydney Cornelius's Wormz Art Studio site. My assigned page was the About page, which included Sydney's biography, contact information, and commission guidelines. For Project 13, I worked under Avery Martin on the Rexburg F4 site, and my assigned page was the Tracks page.

Communication as a junior developer was mostly about understanding what page I was responsible for, getting access to the repositories, and using the existing site plan or shared CSS instead of making my page feel separate from the rest of the site. I generally knew what was expected of me because the assigned page names and content were clear.

I was able to contribute meaningful pages to both sites. On the Wormz Art Studio site, I built out the About page with real content and images so it matched the purpose of the art portfolio. On the Rexburg F4 site, I styled the Tracks page with the lead developer's CSS and adjusted it so it fit the site better instead of feeling like a completely different design.

The biggest GitHub issue I ran into was making sure I was working from the latest version before pushing. I learned to pull first, check what changed, and avoid pushing older files over someone else's newer work. That was especially important because each project had multiple students editing the same repository.

General Experience and Reflections

The most challenging part of this project was not the HTML or CSS by itself. The harder part was coordinating files across different people and repositories. A page could look good locally, but if someone pushed an older version or forgot to use the shared stylesheet, the live site could quickly become inconsistent. I was able to overcome that by checking GitHub status often, pulling before working, and communicating more directly with teammates.

What I enjoyed most was seeing the sites start as wireframes and then become real pages. It was satisfying to take a client's rough plan and turn it into a page with images, navigation, spacing, and actual content. I also liked the junior developer work because it made me practice fitting my page into somebody else's design instead of controlling everything myself.

If I could do the project again, I would ask for clearer checkpoints from the beginning. I would want each team member to push smaller updates earlier so problems with links, images, or styles show up before the final week. I would also document the shared CSS classes better for junior developers so everyone knows what patterns to reuse.

I do feel more confident working on real-world projects after this experience. I practiced semantic HTML, CSS layout, responsive styling, image management, GitHub Pages publishing, and Git workflow. More importantly, I learned that web development is not just writing code; it is also planning, communicating, reviewing changes, and protecting the work that is already finished.
